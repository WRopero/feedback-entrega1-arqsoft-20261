## 2. Modelos de Django (Backend)

**Lo que esta bien:**
- Modelo `Usuario` extiende `AbstractUser` con email como campo de autenticacion (elimina `username`)
- UUIDs como primary keys en todos los modelos
- `UsuarioManager` personalizado con `create_user` y `create_superuser`
- Relacion `Perfil` OneToOne con `Usuario` con `related_name='perfil'`
- `Habilidad` como M2M a traves de `Perfil` con `related_name='perfiles'`
- `Experiencia` vinculada a `Perfil` con cascade delete y ordering por `-fecha_inicio`
- Enum `TipoUsuario` con `TextChoices` (freelancer, cliente, ambos)
- `PublicacionManager` con metodos `activas()` y `por_categoria()` - buen uso de custom managers
- Indices en `Publicacion`: `(estado, -created_at)` y `categoria`
- `ImagenPublicacion` con `upload_to` dinamico y campo `orden`
- `JSONField` para `incluye` - apropiado para listas de texto variable
- Todos los modelos tienen `__str__`, `verbose_name` y `verbose_name_plural`

**Que mejorar:**

### 2.1 Campo `imagen_url` redundante con `ImagenPublicacion`
`publicaciones/models.py` linea 37: `imagen_url = models.URLField(blank=True)` coexiste con el modelo `ImagenPublicacion`. El frontend en `adapters.ts` linea 31 tiene que hacer fallback entre ambos: `item.imagenes?.[0]?.url || item.imagen_url`. Deben usar una sola fuente de imagenes.

### 2.2 Metodo `actualizar_perfil` nunca usado
`usuarios/models.py` lineas 88-91: el metodo `actualizar_perfil` en `Perfil` no se usa en ningun lugar del codigo. El serializer maneja las actualizaciones directamente. Es codigo muerto.

### 2.3 Funcionalidades core ausentes en `main`
El proyecto se llama "Marketplace de Servicios Profesionales" pero en la rama `main` faltan modelos fundamentales:
- **Contratos/Propuestas**: No hay modelo de contratacion. El frontend de contratos usa datos mock hardcodeados (`features/contracts/data.ts`)
- **Resenas**: No hay modelo de resenas. Los componentes `ReviewCard.tsx` y `ReviewsSummary.tsx` existen pero `rating` y `reviews` siempre son 0 (`adapters.ts` lineas 25-26)
- **Chat/Mensajeria**: No hay modelos de mensajeria en `main`

**Nota:** La rama `feat/chat-contratos` tiene implementados modelos de `Propuesta`, `Contrato`, `Conversacion` y `Mensaje` con buena calidad de codigo, pero **nunca fue mergeada a `main`**. 19 commits con ~3,365 lineas de codigo funcional que no llegan al producto final. Esto es un problema serio de integracion.

## 3. API REST (Views y Serializers)

**Lo que esta bien:**
- Uso correcto de `generics.ListCreateAPIView` y `RetrieveUpdateDestroyAPIView` en publicaciones
- Permiso personalizado `EsCreadorOSoloLectura` para autorizacion a nivel de objeto
- Paginacion con `PageNumberPagination` (page_size=20, max=100)
- Filtros por busqueda, categoria, rango de precio y ordenamiento
- `select_related` y `prefetch_related` para optimizacion de queries (previene N+1)
- Patron soft delete en `destroy()` - cambia estado a 'cerrado' en vez de eliminar
- Serializers diferenciados: `List` vs `Detail` vs `Create`
- `RegistroSerializer` con `@transaction.atomic` para crear usuario + perfil atomicamente
- `habilidad_ids` write-only con `habilidades` read-only - separacion limpia
- Upload de imagenes con limite de 3 por publicacion

**Que mejorar:**

### 3.1 Parametros de precio sin validacion
`publicaciones/views.py` lineas 52-58: los query params `precio_min` y `precio_max` se pasan directamente al filtro sin validar tipo numerico. Enviar un string no numerico causa un `ValueError` no manejado (error 500).

### 3.2 `PerfilView.put()` usa `partial=True` que es semantica de PATCH
`usuarios/views.py` linea 42: el metodo `put` establece `partial=True`, lo cual es semantica de PATCH, no de PUT.

### 3.3 `ExperienciaDetailView` usa APIView manual en vez de generics
`usuarios/views.py` lineas 74-99: maneja manualmente `try/except DoesNotExist`. Usar `generics.RetrieveUpdateDestroyAPIView` seria mas limpio y DRY.

### 3.4 Validacion de password incompleta en registro
`usuarios/serializers.py` linea 9: solo valida `min_length=8`. Los `AUTH_PASSWORD_VALIDATORS` de Django estan configurados en settings pero no se aplican en el serializer de registro.

### 3.5 No hay DEFAULT_PERMISSION_CLASSES
`core/settings.py`: `REST_FRAMEWORK` no define `DEFAULT_PERMISSION_CLASSES`. El default de DRF es `AllowAny`, lo que significa que cualquier nueva view sin permisos explicitos queda abierta.

## 4. Frontend (Next.js + TypeScript)

**Lo que esta bien:**
- Arquitectura basada en features: `features/auth`, `features/services`, `features/profile`
- Cada feature tiene su propio `models.ts`, `services/`, `hooks/`, `components/`
- Patron adapter (`adapters.ts`) para transformar respuestas de API a props de componentes
- Custom hooks (`usePerfil`, `usePublicaciones`, `usePublicacion`, `useExperiencias`, `useHabilidades`)
- `AuthProvider` con `createContext` limpio: `login`, `logout`, `refreshPerfil`
- Next.js App Router con grupo de rutas `(protected)` usando guard de auth en layout
- shadcn/ui para componentes de UI consistentes
- Biome para linting/formatting

**Que mejorar:**

### 4.1 No se usa el refresh token
`AuthContext.tsx`: el login recibe `access` y `refresh` tokens, pero solo guarda `access` en localStorage. El `refresh` token se descarta. Cuando el access token expira (1 hora), el usuario es deslogueado silenciosamente. El endpoint `/api/auth/refresh/` existe pero nunca se llama.

### 4.2 Contratos y Dashboard usan datos hardcodeados
- `features/contracts/data.ts`: 4 contratos mock con datos en ingles
- `features/dashboard/dashboard.tsx` lineas 14-72: `SERVICES` y `CONTRACTS` son constantes hardcodeadas
- `RecommendedServices.tsx` linea 86: `<Link href="/services/1">` hardcodea el link para TODOS los servicios

### 4.3 Inconsistencia de idioma
Los contratos estan completamente en ingles mientras el resto de la app esta en espanol:
- `ContractHeader.tsx`: "Active Contracts", "Review and manage...", "+ Post a Job"
- `ContractFilters.tsx`: "Search by project, client, or keyword..."
- `ContractCard.tsx`: "Client", "View Details"

Esto sugiere que la seccion de contratos fue copiada de un template sin traducir.

### 4.4 Filtros no funcionales en ServiceFilters
`components/ServiceFilters.tsx` lineas 101-121: las secciones "Tiempo de Entrega" y "Calificacion" se renderizan pero no llaman `onFilterChange` y no tienen soporte en el backend.

### 4.5 Uso de `<img>` en vez de `<Image>` de Next.js
`ServiceCard.tsx`, `ServiceGallery.tsx`, `CreateServiceForm.tsx`: usan tags `<img>` directamente, perdiendo optimizacion de imagenes de Next.js.

## 7. Analisis SOLID

### SRP (Single Responsibility Principle)

**Lo que esta bien:**
- Las apps de Django estan bien separadas por dominio: `usuarios` (auth + perfiles) y `publicaciones` (servicios). Cada app tiene una responsabilidad clara.
- Serializers separados por operacion (list/detail/create) - cada uno tiene una sola razon de cambio.
- Frontend con feature-based architecture: cada feature maneja su propio dominio.

**Violaciones:**
1. **`usuarios/views.py` mezcla 4 responsabilidades** en un solo archivo: registro, perfil, habilidades y experiencias. Deberia separarse:

```
usuarios/views/
    __init__.py
    auth.py          # RegistroView
    perfil.py        # PerfilView
    habilidades.py   # HabilidadListView
    experiencias.py  # ExperienciaListCreateView, ExperienciaDetailView
```

2. **`AuthContext.tsx` mezcla autenticacion y datos de perfil.** Login, logout, token management Y carga de perfil viven en el mismo contexto. El perfil deberia tener su propio provider.

3. **`PublicacionCreateSerializer.create()`** crea la publicacion Y crea las imagenes asociadas. La creacion de imagenes deberia delegarse a un servicio o al modelo.

### OCP (Open/Closed Principle)

**Lo que esta bien:**
- `TextChoices` para estados y categorias permite agregar nuevos valores sin modificar logica existente.
- `PublicacionManager` con metodos como `activas()` y `por_categoria()` permite agregar filtros sin modificar vistas.

**Violaciones:**
1. **Categorias hardcodeadas como `TextChoices`** en el modelo `Publicacion`:

```python
class Categoria(models.TextChoices):
    DISENO = 'diseno', 'Diseño Gráfico'
    DESARROLLO = 'desarrollo', 'Desarrollo Web'
    # Para agregar una nueva categoría hay que modificar el modelo
```

Deberia ser un modelo `Categoria` separado con `ForeignKey`, asi se agregan categorias sin tocar codigo:

```python
class Categoria(models.Model):
    nombre = models.CharField(max_length=100, unique=True)
    slug = models.SlugField(unique=True)

class Publicacion(models.Model):
    categoria = models.ForeignKey(Categoria, on_delete=models.PROTECT)
```

2. **`CATEGORIA_LABELS` hardcodeado en `adapters.ts`** duplica las categorias del backend. Si se agrega una categoria en el backend, hay que modificar el frontend tambien.

### LSP (Liskov Substitution Principle)

**Lo que esta bien:**
- `Usuario` extiende `AbstractUser` correctamente, manteniendo el contrato de Django auth.
- `EsCreadorOSoloLectura` extiende `BasePermission` respetando la interfaz.

**Sin violaciones evidentes** - los modelos no tienen herencia profunda que pueda romper LSP.

### ISP (Interface Segregation Principle)

**Violaciones:**
1. **`PerfilSerializer` es un serializer gordo** con 13 campos que maneja lectura, escritura, foto, habilidades y experiencias todo junto. Un cliente que solo necesita el nombre y email recibe todo el payload. Deberian existir:
   - `PerfilResumenSerializer` (nombre, email, foto)
   - `PerfilCompletoSerializer` (todo)
   - `PerfilUpdateSerializer` (campos editables)

2. **En el frontend hay dos interfaces `Perfil` diferentes** (`features/auth/models.ts` y `features/profile/models.ts`). El de auth es un subset del de profile. Deberian compartir una interfaz base y extenderla.

### DIP (Dependency Inversion Principle)

**Violaciones criticas:**
1. **No hay capa de servicios.** Las vistas llaman directamente a los modelos. No hay abstraccion intermedia. `RegistroView` crea el usuario directamente via el serializer. `PublicacionListCreateView` filtra directamente el ORM. Si manana necesitan cambiar la logica de registro (agregar email de verificacion, notificaciones), hay que modificar la vista.

```python
# Actual (vista acoplada al ORM)
class RegistroView(APIView):
    def post(self, request):
        serializer = RegistroSerializer(data=request.data)
        serializer.save()  # Directo al modelo

# Correcto (vista delega a servicio)
class RegistroView(APIView):
    def post(self, request):
        serializer = RegistroSerializer(data=request.data)
        serializer.is_valid(raise_exception=True)
        RegistrationService.register_user(serializer.validated_data)
```

2. **En la rama `feat/chat-contratos`, `PropuestaDetailView.put()` importa `chat.models` dentro del metodo:**

```python
def put(self, request, pk):
    ...
    if nuevo_estado == Propuesta.Estado.ACEPTADA:
        from chat.models import Conversacion  # Import directo a clase concreta
        Conversacion.objects.create(contrato=contrato)
```

Esto acopla contratos directamente a chat. Deberia usar un servicio o senales de Django:

```python
# Usando signals (desacoplado)
from django.db.models.signals import post_save
from django.dispatch import receiver

@receiver(post_save, sender=Contrato)
def crear_conversacion_para_contrato(sender, instance, created, **kwargs):
    if created:
        Conversacion.objects.create(contrato=instance)
```
